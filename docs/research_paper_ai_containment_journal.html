National Security AI-Containment System: A Multi-Layer Architecture for Behavioral Monitoring, Threat Detection and Fail-Closed Mediation of Large Language Models

Aryan Pathania, Aryan Sharma
CGC University, Mohali, Punjab, India
pathaniaaryan169@gmail.com, aryansharma2004march@gmail.com

Abstract: Large language models are increasingly attractive for defense-adjacent workflows such as intelligence triage, cyber analysis, secure knowledge access and decision support. However, the same instruction-following behavior that makes these systems useful also exposes them to prompt injection, malicious code generation, data exfiltration attempts, unsafe network actions and policy-violating outputs. This paper presents a deployable AI-containment prototype implemented in Django and centered on a multi-layer mediation pipeline named Sentinel. The architecture combines an isolated model-access layer, a hybrid threat-detection engine, a policy-governed response layer, and a real-time command dashboard with alerting, forensic logging and an emergency kill switch. The detection stack fuses lexical rules, regex signatures, semantic similarity scoring, Shannon-entropy analysis and an optional supervised classifier trained on a 137,000-record eight-class dataset assembled from public jailbreak, benign dialogue and harmful-instruction corpora. On a held-out validation split, the classifier achieved 98.26% accuracy. Operational logs from the implemented prototype recorded 176 interactions, of which 83 were flagged as suspicious or threatening, 34 reached critical severity, 42 generated kill-switch trigger events and 127 produced real-time alerts. Prompt injection, malicious code and data exfiltration were the dominant categories. Results indicate that layered containment with fail-closed mediation can make unsafe model behavior substantially more observable, auditable and interruptible even when the underlying language model remains vulnerable to jailbreaks. The contribution of this work is not a claim of perfect containment, but a practical engineering architecture that integrates detection, enforcement, operator visibility and emergency shutdown into a single security workflow.

Keywords: AI containment, large language models, prompt injection, threat detection, kill switch, national security, secure deployment, human-in-the-loop monitoring.

I. INTRODUCTION

Large language models (LLMs) are moving quickly from experimental assistants to embedded decision-support components in high-impact environments. Within national-security and critical-infrastructure contexts, the attraction is obvious: a single model can summarize long reports, classify incidents, propose code, triage alerts and draft responses at a speed that reduces operator load. Yet the same broad generative ability creates a difficult trust problem. A system that can reason over natural language instructions is also a system that can be manipulated by hostile instructions, induced to reveal sensitive information, or steered toward dangerous outputs that appear operationally useful at first glance.

The containment problem is therefore not solved merely by selecting an aligned model provider. Oracle-AI and capability-control literature has long argued that limiting the operational surface of a powerful system is at least as important as shaping its internal objective function [1]. Modern deployment guidance follows the same logic. NIST's AI RMF 1.0 and the Generative AI Profile emphasize governance, measurement and deployment-time risk controls rather than treating model alignment as a complete answer [2], [3]. Likewise, joint guidance from CISA and NSA argues that externally developed AI systems should be surrounded by defensive controls that preserve confidentiality, integrity and availability during operation [4].

Recent jailbreak research reinforces why this layered posture is necessary. Prompt injection can override intended instructions in both direct and indirect settings [5], universal adversarial suffixes can transfer across aligned models [6], and long-context many-shot jailbreaking can cause increasingly harmful completions as demonstrations accumulate [7]. OWASP has accordingly framed prompt injection as a first-class application vulnerability, not merely a model oddity [8]. Put plainly, the application boundary around an LLM is now part of the security perimeter.

This work presents a research-style reconstruction of a functioning prototype developed in the ai_containment workspace. The system is not a theoretical framework alone; it includes backend mediation, configurable policies, a hybrid detector, real-time alerts, audit persistence and an operator-facing command center. Its purpose is to treat unsafe model interaction as an observable and governable process rather than a hidden model-side failure.

The core contributions of the paper are fourfold. First, it defines a concrete three-layer containment architecture that separates model access, threat mediation and operator oversight. Second, it implements a threat detector that combines symbolic features with semantic and supervised components rather than relying on a single heuristic. Third, it operationalizes enforcement through fail-closed blocking, policy-based escalation and a kill switch that can suspend further model generation. Fourth, it validates the approach using both offline classifier metrics and live operational logs collected from the deployed prototype.

The system should not be interpreted as a final answer to general AI safety. Instead, it should be understood as a security engineering pattern for present-day LLM deployments in which the model is treated as potentially misbehaving software that must be mediated, monitored and, when necessary, interrupted. That framing is especially useful in national-security settings, where the cost of a single unsafe completion may exceed the convenience gained from permissive model access.

II. RELATED WORK

Early containment thinking focused on restricting the external action surface of an advanced intelligence system. Armstrong, Sandberg and Bostrom described Oracle AI as a system that only answers questions rather than acting directly in the world, while also showing that even this narrow configuration remains risky if operators over-trust answers or expose the system to poorly constrained interaction channels [1]. The current work inherits that intuition but translates it into a practical web application pattern suitable for modern LLM deployment.

Governance frameworks have since expanded that intuition into lifecycle guidance. NIST AI RMF 1.0 organizes risk management around the functions Govern, Map, Measure and Manage, while the 2024 Generative AI Profile extends those ideas to model-specific failure modes such as misuse, data leakage and operational unreliability [2], [3]. The architecture proposed here aligns strongly with that deployment view: the model is only one component, and assurance depends on the controls wrapped around it.

Prompt-injection research demonstrates why deployment-time control cannot be optional. Greshake et al. showed that application-integrated LLMs are vulnerable not only to direct malicious user input but also to indirect prompt injection delivered through external content such as retrieved web pages and documents [5]. Zou et al. later demonstrated universal and transferable adversarial suffixes that can systematically induce harmful behavior across aligned models [6]. Anthropic's many-shot jailbreaking results further show that longer context windows can increase exposure to attack rather than merely increasing utility [7]. These findings suggest that security-sensitive deployments need pre- and post-generation scrutiny rather than a single trust decision at model selection time.

Operational security guidance from OWASP and CISA pushes the same conclusion from the application side. OWASP recommends structured prompts, input filtering, output validation, least privilege, monitoring and emergency controls including kill switches [8]. CISA's deployment guidance similarly emphasizes layered mitigations, vulnerability response, logging and response procedures [4]. The present system can be viewed as a prototype implementation that turns these recommendations into a concrete control plane for AI interaction.

The main gap in prior work is that many publications either analyze attacks abstractly or propose defenses in isolation. Fewer works describe a full-stack containment prototype that joins model mediation, detection, dashboarding, logging, emergency shutdown and policy management in one deployable application. That integration gap is the problem addressed by Sentinel.

III. PROPOSED SYSTEM ARCHITECTURE

A. Layer 1: Contained model access

The first layer standardizes communication with the underlying language model. In the current implementation this layer supports three backends: a deterministic mock engine for demonstrations, a local Ollama backend, and an OpenAI-compatible backend. The goal is not merely convenience. By forcing all model calls through a single orchestration class, the system ensures that a common system prompt, token budget and kill-switch state are enforced before any completion is requested.

This layer also embodies a minimum-privilege philosophy. The model receives the user prompt and a fixed operational instruction, but it does not directly control alerting, policy writes, dashboard state or audit persistence. Those actions are handled elsewhere in the application. This separation reduces the chance that a model-side misbehavior can translate immediately into system-side authority.

B. Layer 2: Hybrid Sentinel detector

The second layer is the core containment engine. It analyzes both user input and model output using a configurable knowledge base of threat categories such as prompt injection, malicious code, data exfiltration, obfuscation, weapons and network access. Each category contains severity metadata, auto-kill behavior, keywords, regex patterns and semantic anchor phrases. The engine merges evidence from multiple detectors into a category confidence score and escalates overall threat level according to the most severe matched behavior.

Policy handling is dynamic. Category severity and auto-kill behavior can be overridden through stored security policies, allowing operators to tighten or relax response posture without changing source code. This design is important for a national-security framing because deployment context changes over time; a model action that is acceptable in a classroom demo may be unacceptable in a live cyber-defense workflow.

C. Layer 3: Operational command center

The third layer turns containment into a managed process. It provides dashboards for recent interactions, threat distribution, sandbox testing, policy administration, alert review and kill-switch operations. The current Django-based interface uses WebSocket updates to push newly generated alerts and interaction summaries to the frontend in near real time. This means model misuse is visible to operators during use rather than only through later forensic review.

Together, the three layers form a practical mediation pattern: the model is allowed to generate, but only through a gate; the gate is allowed to block, but only according to inspectable evidence and policy; and the operator is allowed to override or investigate, but only with an auditable trail. That is the containment principle implemented in this prototype.

IV. THREAT MODEL AND MEDIATION PIPELINE

The prototype assumes an attacker may act as an external user, an insider tester, or an indirect content source whose text is eventually processed by the model. The threats of immediate concern are direct prompt injection, model-assisted malicious code generation, credential and data exfiltration, hidden payloads, unsafe network instructions and general policy-violating requests. The present system does not claim resistance against every advanced adversary; instead it aims to sharply reduce the number of unsafe interactions that pass silently through an application boundary.

The request path begins with input scanning. Every user prompt is analyzed before it reaches the model. If the input is categorized as non-safe, the system follows a fail-closed policy and blocks forwarding. If the category is configured for auto-kill, the global kill switch is triggered immediately and the event is persisted. Safe input is then sent to the contained model, after which the raw model output is analyzed by the same Sentinel engine before the response is released to the user.

The detector combines four normalized evidence channels: keyword hits, regex hits, semantic similarity and entropy anomaly. An optional supervised classifier provides a fifth signal by predicting the most likely threat class from the text. The configured score weights in the implemented system are 0.30 for keyword evidence, 0.30 for regex evidence, 0.30 for semantic evidence and 0.10 for entropy. A category is emitted when the combined score exceeds 0.25, after which severity and blocking behavior are determined by policy metadata.

score(c) = 0.30K + 0.30R + 0.30S + 0.10E,   emit c if score(c) >= 0.25

Threat aggregation also captures combinations that are operationally more serious than their parts. For example, the engine raises the final level to critical when prompt injection and data exfiltration signals co-occur. This design recognizes that many realistic AI misuse attempts are chained behaviors rather than isolated labels: an attacker first seeks to override instructions, then to retrieve or transmit sensitive data.

Fig. 2. Request mediation workflow. The same detector evaluates both inbound prompts and outbound model responses, while critical matches can trigger an emergency shutdown path.

A key design choice is that output scanning remains active even when the input was safe. This is necessary because a benign user request can still elicit a risky completion from a permissive or compromised model. In the current prototype, several logged incidents arose from this exact pattern, confirming that input filtering alone is insufficient for safe deployment.

V. DATASET ENGINEERING AND MODEL TRAINING

To supplement symbolic detection, the project includes a management workflow for building and training a supervised classifier. The dataset builder assembles a large labeled corpus from public sources spanning direct prompt injection, in-the-wild jailbreak prompts, safe conversational text and harmful-instruction benchmarks. The implemented "large" profile targets eight labels and stores the merged output as a JSONL file for reproducible model training.

The current large-profile corpus contains 137,000 records. Safe examples dominate the distribution to reflect everyday use, but prompt injection and harmful classes remain well represented to support robust separation boundaries. The builder normalizes label names, de-duplicates near-identical text using hashing and balances under-filled categories with synthetic defensive examples when necessary.

The training command builds a TF-IDF word-ngram representation and feeds it to logistic regression with class balancing. In the configuration reflected by the stored metrics file, the model used 1-2 gram features, up to 120,000 terms and an 80/20 stratified validation split. This resulted in 109,600 training samples and 27,400 validation samples. The learned model is persisted as a joblib payload and loaded by the Sentinel engine at runtime whenever the artifact is available.

The choice of logistic regression is deliberate. While more complex encoders could improve recall on nuanced phrasing, the application prioritizes transparent probabilities, fast reloads, lightweight deployment and simple integration with symbolic evidence. In a layered defense architecture, the classifier does not need to carry the entire burden of detection; it serves as one interpretable signal among several.

VI. IMPLEMENTATION DETAILS

The prototype is implemented as a Django application with two main apps: sentinel and dashboard. The sentinel app contains the mediation logic, model wrapper, threat models, management commands and persistent database tables. The dashboard app provides views for command-center analytics, logs, alerts, policy toggling, kill-switch status and an interactive sandbox used to probe the containment pipeline with both safe and adversarial prompts.

Operational persistence is intentionally rich. Every interaction can store a session identifier, input hash, original prompt, raw model output, filtered output, token counts, final threat level, whether the input itself was threatening, whether blocking occurred and whether the kill switch was triggered. Threat evidence is stored in a child table that records category, severity, confidence, matched keywords, matched patterns and direction (input or output). Separate tables persist security policies, kill-switch events and operator alerts. This schema turns the application into a lightweight forensic appliance rather than a transient demo.

The live command center aggregates statistics such as threat counts, daily activity, severity distribution and unread alerts. WebSocket-based broadcast pushes critical state changes to the UI, allowing analysts to watch the pipeline as it executes. The sandbox view is also important from a research perspective because it enables structured red-team style evaluation of the same path that end users would traverse in production.

The implementation currently uses SQLite and an in-memory channel layer, which is sufficient for a prototype but not the intended scaling endpoint. A production deployment would likely move to durable message infrastructure, stronger operator authentication, hardened admin workflows, and host-level isolation around any locally executed tools. These limitations are acknowledged because containment claims should match implementation reality.

VII. EXPERIMENTAL RESULTS

A. Offline classifier evaluation

The stored training metrics show that the supervised component reached 98.26% validation accuracy on the held-out split. Performance was especially strong on data exfiltration, malicious code, network access, obfuscation and weapons categories, each of which achieved near-perfect precision and recall. Prompt injection remained the most difficult label, with recall lower than the other harmful classes, which is unsurprising given the linguistic diversity of jailbreak phrasing.

B. Online operational behavior

Operational evidence was drawn from the prototype's SQLite audit database. The current log set contains 176 total interactions. Of these, 83 were marked suspicious or threatening by the mediation pipeline, and 36 were stopped at the input stage before any model completion was generated. Importantly, 18 additional incidents began with inputs classified as safe but were later flagged after output scanning, which directly demonstrates the value of dual-sided mediation.

The system also recorded 42 kill-switch trigger events and 127 real-time alerts. Seven active policies were present in the database at the time of inspection. Across all threat-detail records, input-side findings outnumbered output-side findings 106 to 59, indicating that many direct attack attempts are stopped early. Nevertheless, the output-side findings included malicious code, prompt-injection remnants, unsafe network instructions, data-exfiltration content and smaller counts of broader policy-violation categories, confirming that post-generation review remains necessary.

The mean logged analysis time was approximately 3.09 s, while average recorded prompt and completion lengths were 64.10 and 164.94 tokens, respectively. This latency is acceptable for an operator-mediated security prototype but leaves room for optimization, especially if semantic models are upgraded or if higher interaction rates are expected. In a stricter production scenario, asynchronous triage or tiered detection could reduce delay for obviously safe inputs.

The most important qualitative result is not raw accuracy alone. It is the demonstrated ability to convert unsafe model interaction into a sequence of governed states: detected, blocked, redacted, logged, alerted and, when necessary, halted. From a security operations perspective, that state transition matters more than the specific classifier family chosen.

VIII. DISCUSSION

The prototype shows that containment is most credible when it is treated as an application-architecture problem rather than a single-model property. Because the detector scans both inputs and outputs, the system can catch two distinct failure modes: overtly adversarial prompting and benign-looking prompts that nevertheless lead to harmful generations. The audit schema, alerting path and kill-switch workflow further ensure that the response to unsafe behavior is not invisible. This is especially valuable in environments where after-action investigation and operator accountability are mandatory.

At the same time, the system has clear limits. It does not provide cryptographic assurance that a sufficiently capable model cannot manipulate downstream users. It does not yet isolate tool execution in a real operating-system sandbox, nor does it handle multimodal prompt injection. The supervised dataset is partly benchmark-driven and partly synthetically balanced, which helps coverage but may not perfectly reflect field data. The operational log set also contains intentional adversarial testing, so the observed threat rate should be interpreted as evidence of mediation behavior rather than a claim about real-world base rates.

These limitations do not negate the value of the architecture. Rather, they clarify where future work should focus: better indirect-content sanitation, richer behavioral policies, externalized message queues, stronger admin control, multimodal scanning, red-team automation and integration with host-level execution sandboxes. In short, containment should evolve alongside the model's capability surface.

IX. CONCLUSION AND FUTURE WORK

This paper presented Sentinel, a multi-layer AI-containment prototype for security-sensitive LLM deployment. The system combines contained model access, a hybrid threat detector, dynamic policy enforcement, operator visibility, forensic logging and an emergency kill switch. The supervised component trained on a 137,000-record dataset achieved 98.26% validation accuracy, while operational logs from the deployed prototype showed that the architecture can detect, block and escalate a meaningful variety of unsafe interactions in practice.

The principal takeaway is that deployable containment does not require solving general AI safety in one step. It requires building concrete mediation points where risk can be measured and controlled. For national-security use cases, such mediation is not optional overhead; it is part of the minimum trustworthy deployment boundary. Future work should extend the current design toward multimodal input inspection, more formal policy languages, distributed audit pipelines and stronger runtime isolation around tool use. Even in its present form, however, the prototype demonstrates that layered containment can convert a vulnerable language model from an opaque liability into a monitorable and interruptible subsystem.
